# PRD: Catálogo Digital WhatsApp Commerce SaaS

**Producto:** WhatsCatalog Pro  
**Tecnologías Clave:** Next.js 14, Supabase (Auth + PostgreSQL + Storage), Tailwind CSS, shadcn/ui, WhatsApp API (wa.me links)

## 1. Visión del Producto

Transformar la gestión de inventario y ventas informales en una experiencia digital profesional y sin fricción. WhatsCatalog Pro permite a pequeños y medianos comerciantes crear catálogos digitales elegantes en minutos, gestionar su inventario inteligentemente y recibir pedidos directamente por WhatsApp sin necesidad de desarrollar una tienda online compleja ni pagar comisiones por venta.

El producto actúa como el puente perfecto entre el control de stock empresarial y la simplicidad de las ventas por WhatsApp que ya dominan los comerciantes latinoamericanos.

### 3.4 Sistema de Carrito y Checkout WhatsApp

**Lógica del Carrito (Cliente-Side):**
- Persistencia en `localStorage` (no requiere registro del comprador)
- Estructura: `[{producto_id, cantidad, nombre, precio_unitario, imagen}]`
- Sincronización entre pestañas (BroadcastChannel API)
- Badge en botón flotante con cantidad total de items

**Flujo de Pedido:**

**Paso 1: Revisión del Carrito**
- Lista de productos con miniaturas
- Cantidad editable por producto
- Botón eliminar (icono trash)
- Subtotal calculado automáticamente
- Botón "Continuar con el Pedido"

**Paso 2: Formulario de Datos (Modal/Drawer)**

| Campo | Tipo | Validación | Requerido |
|-------|------|------------|-----------|
| Nombre completo | Text | Mínimo 3 caracteres | Sí |
| Teléfono | Tel | Formato válido por país | Sí |
| Dirección | Textarea | Mínimo 10 caracteres | Sí |
| Método de Pago | Radio Cards | Solo una selección | Sí |
| Método de Entrega | Radio Cards | Delivery o Pickup | Sí |
| Notas adicionales | Textarea | Opcional | No |

**Métodos de Pago (Tarjetas Option Button):**
- 💳 Transferencia bancaria
- 📱 Pago móvil / PSE / Mercado Pago (según país)
- 💵 Efectivo / Divisas
- (Personalizable por el vendedor en configuración)

**Métodos de Entrega:**
- 🏠 Delivery (muestra costo si aplica)
- 🏪 Pickup / Retiro en tienda (muestra dirección del negocio)

**Validaciones en Tiempo Real:**
- Campos con borde rojo si son inválidos
- Botón "Enviar Pedido por WhatsApp" deshabilitado hasta validación completa
- Mensajes de error específicos por campo

**Paso 3: Generación de Mensaje WhatsApp**

**Plantilla dinámica generada:**

```
¡Hola! 👋 Soy *{nombre_cliente}* y quiero hacer un pedido:

📦 *PRODUCTOS:*
{lista_productos_con_precios}

💰 *TOTAL: ${monto_total}*

📍 *ENTREGA:* {metodo_entrega}
📱 *TELÉFONO:* {telefono}
🏠 *DIRECCIÓN:* {direccion}

💳 *MÉTODO DE PAGO:* {metodo_pago}
📝 *NOTAS:* {notas_opcionales}

¿Está disponible todo para confirmar mi pedido? Gracias! 🙏
```

**Acción Final:**
- URL encode del mensaje
- Redirección a: `https://wa.me/{numero_negocio}?text={mensaje_encoded}`
- Apertura en nueva pestaña (target="_blank")
- Limpieza automática del carrito post-redirección

### Esquema de Base de Datos (PostgreSQL):

```sql
-- Tenants (Negocios)
CREATE TABLE tenants (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    nombre_negocio VARCHAR(255) NOT NULL,
    slug VARCHAR(100) UNIQUE NOT NULL, -- para subdominio
    whatsapp VARCHAR(20) NOT NULL,
    moneda VARCHAR(3) DEFAULT 'USD',
    color_primario VARCHAR(7) DEFAULT '#3B82F6',
    color_secundario VARCHAR(7) DEFAULT '#10B981',
    logo_url TEXT,
    banner_url TEXT,
    mensaje_bienvenida TEXT,
    costo_delivery DECIMAL(10,2) DEFAULT 0,
    activo BOOLEAN DEFAULT true,
    created_at TIMESTAMP DEFAULT NOW()
);

-- Usuarios (Vendedores/Admins)
CREATE TABLE profiles (
    id UUID PRIMARY KEY REFERENCES auth.users(id),
    tenant_id UUID REFERENCES tenants(id),
    nombre VARCHAR(100),
    rol VARCHAR(20) DEFAULT 'admin', -- admin, vendedor
    avatar_url TEXT
);

-- Categorías
CREATE TABLE categorias (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id UUID REFERENCES tenants(id),
    nombre VARCHAR(100) NOT NULL,
    icono VARCHAR(50),
    orden INTEGER DEFAULT 0,
    destacada BOOLEAN DEFAULT false
);

-- Productos
CREATE TABLE productos (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id UUID REFERENCES tenants(id),
    categoria_id UUID REFERENCES categorias(id),
    nombre VARCHAR(255) NOT NULL,
    descripcion TEXT,
    precio DECIMAL(10,2) NOT NULL,
    stock INTEGER DEFAULT 0,
    stock_minimo INTEGER DEFAULT 5,
    sku VARCHAR(100),
    imagen_principal TEXT,
    imagenes_adicionales JSONB DEFAULT '[]',
    estado VARCHAR(20) DEFAULT 'activo', -- activo, inactivo, agotado
    vistas INTEGER DEFAULT 0,
    created_at TIMESTAMP DEFAULT NOW()
);

-- Movimientos de Inventario (Auditoría)
CREATE TABLE movimientos_stock (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    producto_id UUID REFERENCES productos(id),
    tenant_id UUID REFERENCES tenants(id),
    tipo VARCHAR(20), -- entrada, salida, ajuste
    cantidad INTEGER NOT NULL,
    stock_anterior INTEGER,
    stock_nuevo INTEGER,
    motivo TEXT,
    usuario_id UUID REFERENCES profiles(id),
    created_at TIMESTAMP DEFAULT NOW()
);
```

### Diagrama de Flujo Lógico:

```
1. [Tenant Detection]
   ↓ Middleware detecta subdominio/dominio
   ↓ Busca tenant_id en DB
   ↓ Inyecta contexto en request

2. [Catálogo Público]
   ↓ Fetch productos WHERE tenant_id = X AND estado = 'activo'
   ↓ Agrupa por categorías ordenadas
   ↓ Render SSR con metadatos dinámicos (Open Graph)

3. [Interacción Producto]
   ↓ Click → Modal con detalle (Client Component)
   ↓ Add to Cart → localStorage + Zustand
   ↓ Cart persistido entre sesiones

4. [Checkout]
   ↓ Validación formulario (Zod schema)
   ↓ Generación mensaje WhatsApp (template engine)
   ↓ wa.me link con texto prellenado
   ↓ Redirect + Clear cart
```

### Arquitectura de Seguridad:

- **RLS Policies:** Cada tabla tiene políticas que filtran por `tenant_id`
- **Storage:** Buckets separados por tenant, políticas de lectura pública solo para imágenes de productos activos
- **Rate Limiting:** 100 requests/minuto por IP en catálogo público
- **Validación:** Todas las inputs validadas con Zod antes de tocar DB

## 5. Especificaciones Técnicas para el Ingeniero (Actualizado)

### 5.1 Esquema de Base de Datos Normalizado (PostgreSQL)

He reestructurado completamente el esquema para eliminar redundancias y dependencias transitivas, aplicando rigurosamente las tres formas normales.

#### **Análisis de Normalización:**

**Primera Forma Normal (1FN):** Todos los atributos contienen valores atómicos indivisibles. No hay grupos repetitivos.

**Segunda Forma Normal (2FN):** Cumple 1FN y todos los atributos no clave dependen completamente de la clave primaria (elimina dependencias parciales).

**Tercera Forma Normal (3FN):** Cumple 2FN y no existen dependencias transitivas (atributos no clave no dependen de otros atributos no clave).

---

#### **Tablas del Esquema:**

```sql

-- ==========================================
-- 10. PEDIDOS (Registro histórico - nueva tabla para trazabilidad)
-- ==========================================
CREATE TABLE pedidos (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id UUID REFERENCES tenants(id) ON DELETE CASCADE,
    
    -- Datos del cliente (denormalizado intencionalmente para histórico inmutable)
    cliente_nombre VARCHAR(200) NOT NULL,
    cliente_telefono VARCHAR(20) NOT NULL,
    cliente_email VARCHAR(255),
    cliente_direccion TEXT NOT NULL,
    
    -- Métodos seleccionados (snapshots en el momento del pedido)
    metodo_pago_id UUID REFERENCES metodos_pago_config(id),
    metodo_pago_snapshot JSONB, -- Datos completos del método elegido
    
    opcion_entrega_id UUID REFERENCES opciones_entrega(id),
    opcion_entrega_snapshot JSONB,
    
    -- Totales
    subtotal DECIMAL(12,2) NOT NULL,
    costo_envio DECIMAL(12,2) DEFAULT 0,
    descuento DECIMAL(12,2) DEFAULT 0,
    total DECIMAL(12,2) NOT NULL,
    
    -- Estado del pedido
    estado VARCHAR(20) DEFAULT 'pendiente' CHECK (estado IN ('pendiente', 'confirmado', 'preparando', 'enviado', 'entregado', 'cancelado')),
    notas_cliente TEXT,
    
    -- Metadata WhatsApp
    whatsapp_mensaje_enviado TEXT,
    whatsapp_enviado_at TIMESTAMP,
    
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

-- ==========================================
-- 11. DETALLE_PEDIDO (Items individuales - evita arrays en pedidos)
-- ==========================================
CREATE TABLE detalle_pedido (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    pedido_id UUID REFERENCES pedidos(id) ON DELETE CASCADE,
    producto_id UUID REFERENCES productos(id) ON DELETE SET NULL, -- SET NULL para mantener histórico si se borra producto
    
    -- Snapshot del producto (inmutable)
    producto_nombre VARCHAR(255) NOT NULL,
    producto_sku VARCHAR(100),
    producto_imagen_url TEXT,
    
    cantidad INTEGER NOT NULL CHECK (cantidad > 0),
    precio_unitario DECIMAL(12,2) NOT NULL,
    subtotal DECIMAL(12,2) NOT NULL,
    
    created_at TIMESTAMP DEFAULT NOW()
);

-- ==========================================
-- 12. ANALYTICS Y EVENTOS (Tracking desnormalizado para performance)
-- ==========================================
CREATE TABLE eventos_catalogo (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id UUID REFERENCES tenants(id) ON DELETE CASCADE,
    
    tipo_evento VARCHAR(50) NOT NULL, -- 'vista_producto', 'add_cart', 'inicio_checkout', 'pedido_whatsapp'
    sesion_id VARCHAR(100), -- Para agrupar acciones del mismo visitante
    ip_address INET,
    user_agent TEXT,
    
    -- Datos específicos del evento
    producto_id UUID REFERENCES productos(id),
    metadata JSONB, -- {cantidad: 2, categoria_id: "uuid", origen: "instagram"}
    
    created_at TIMESTAMP DEFAULT NOW()
);

-- Índices optimizados
CREATE INDEX idx_productos_tenant ON productos(tenant_id);
CREATE INDEX idx_productos_categoria ON productos(categoria_id);
CREATE INDEX idx_productos_estado ON productos(estado);
CREATE INDEX idx_movimientos_producto ON movimientos_stock(producto_id, created_at DESC);
CREATE INDEX idx_pedidos_tenant ON pedidos(tenant_id, created_at DESC);
CREATE INDEX idx_eventos_tenant_tipo ON eventos_catalogo(tenant_id, tipo_evento, created_at);
```

### 5.5 Políticas de Seguridad (RLS) Actualizadas

```sql
-- Política: Usuarios solo ven métodos de pago de su tenant
CREATE POLICY "tenant_isolation_metodos_pago" ON metodos_pago_config
    FOR ALL USING (tenant_id IN (
        SELECT tenant_id FROM profiles WHERE id = auth.uid()
    ));

-- Política: Datos bancarios nunca se exponen en API pública
-- El backend filtra campos sensibles (numero_cuenta completo) 
-- antes de enviar al frontend del catálogo público

-- Vista segura para catálogo público (solo datos necesarios)
CREATE VIEW metodos_pago_publicos AS
SELECT 
    id,
    tenant_id,
    titulo_display,
    instrucciones,
    banco_nombre,
    CONCAT('****', RIGHT(numero_cuenta, 4)) as numero_cuenta_mask,
    tipo_cuenta,
    documento_identidad,
    nombre_titular,
    telefono_pago
FROM metodos_pago_config
WHERE activo = true AND mostrar_en_catalogo = true;
```
